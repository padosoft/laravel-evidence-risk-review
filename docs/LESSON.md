# Lessons

## 2026-06-13

- The repository started as a clean stub on `main` with only `.git`, `LICENSE`, and a placeholder `README.md`; bootstrap documentation is the first task before package code.
- The canonical package spec is stored outside the repo at `%USERPROFILE%\Downloads\padosoft-laravel-evidence-risk-review-SPEC-PLAN.md`; keep a durable local plan in `docs/IMPLEMENTATION_PLAN.md` so future sessions are not blocked by missing chat context.
- The prior `product_image_discovery_admin` project established a reliable process pattern: maintain `AGENTS.md`, `docs/RULES.md`, `docs/PROGRESS.md`, `docs/LESSON.md`, and a repo-local skill before implementation starts.
- GitHub Copilot Code Review should be requested as an actual reviewer. If `gh pr edit <PR> --add-reviewer @copilot` fails due to missing `read:project`, use the GraphQL `requestReviewsByLogin` mutation with `copilot-pull-request-reviewer[bot]` and verify requested reviewers. The REST `reviewers[]=copilot` fallback can be misleading.
- Local review must use the full branch diff against `origin/main`, not just uncommitted changes, because Copilot needs the complete context of the task branch.
- The package must enforce standalone neutrality with an architecture test from W1 onward; this is cheaper than trying to remove host coupling later.
- Default-OFF behavior for HTTP API, MCP, and LLM must be tested in the package itself because this package is intended to be safe when installed but not configured.
- On this Windows workstation, prefer Herd PHP for local PHP gates. Validate the exact PHP 8.5 executable/wrapper during W1 before documenting a command as canonical.
- Reference local workstation paths with `%USERPROFILE%` rather than committing the concrete username. Always pair local paths with a fallback instruction ("If unavailable, continue from `docs/IMPLEMENTATION_PLAN.md`") so agents on other machines are not blocked. Apply this fallback to every file that lists the path, including `docs/RULES.md` — it was initially missing there.
- Two SKILL.md files exist for different agent runtimes (`.claude/skills/` for Copilot CLI, `skills/` for Claude Code); they must be kept in sync. Add a note to the shorter one referencing the fuller one to make the relationship explicit.
- `docs/IMPLEMENTATION_PLAN.md` should have a formal `Bootstrap` exit gate section so an agent resuming from context-loss knows when the bootstrap task is considered complete.
- A `.gitignore` should be added in the bootstrap branch—it is the first branch before any application code and the right moment to protect `vendor/`, `.env`, and `*.diff` files.
- The bootstrap milestone is the only one that PRs directly to `main`; mark this as an explicit exception in the milestone heading so future agents do not follow the same pattern for W1+ work.
- The Bootstrap exit gate must include `docs/PROGRESS.md` and `docs/LESSON.md` update steps, matching the generic subtask exit gate template.
- The `copilot --autopilot --yolo` review command can mutate files and even amend commits during a nominal `/review` pass. For review-only gates, use report-only prompting and avoid `--yolo` by default; the prompt must say not to edit files, not to run shell commands, not to stage files, and not to commit. If a timeout happens, record it as blocked and do not push/PR unless the review later completes with zero actionable comments or the user explicitly approves a manual-review exception.
- Copilot CLI can still attempt filesystem edits even when prompted not to. After every local Copilot review, check `git status` and inspect the diff before staging; keep only intentional changes and record the behavior in `docs/PROGRESS.md`.
- Process docs need redundant guardrails at every entry point. If `AGENTS.md` or `IMPLEMENTATION_PLAN.md` changes the Copilot/report-only, CI, branch, or exit-gate rules, mirror the same operational constraint in `docs/RULES.md` and the repo-local skill.
- When a process guardrail is added, propagate its full shape everywhere: command mode, return contract, timeout behavior, post-review `git status`, and push/PR blocking semantics.
- For local Copilot review in this repo, the stable command is stdin without `--autopilot`: write the full diff to a temp file, pipe it with `Get-Content -Raw ... | copilot -p "/review ..."`. The file-path `--autopilot` variant repeatedly timed out and sometimes attempted edits.
- Until W7 creates GitHub Actions, the remote CI gate is "no required checks failing or pending"; W7 must introduce the real workflow and all later PRs must wait for it.
- User clarified the local Copilot gate policy: after three consecutive local Copilot CLI failures for the same subtask (quota, timeout, or CLI failure), record the attempts, proceed with the PR loop, and retry local Copilot on the next macro/subtask. This does not bypass tests, remote GitHub Copilot Code Review, or CI.
- `padosoft-laravel-flow` discovered the GitHub Copilot Code Review engagement loop. With current GitHub CLI, request Copilot at PR creation with `gh pr create --reviewer '@copilot'` whenever possible, or `gh pr edit <PR> --add-reviewer '@copilot'` for existing PRs. Quote `@copilot` in PowerShell. If that fails or no-ops after verification, try `gh pr edit <PR> --add-reviewer copilot-pull-request-reviewer`.
- Verify every Copilot request with `gh api repos/<owner>/<repo>/pulls/<PR>/requested_reviewers` and `gh pr view --json reviewRequests,reviews,comments`; if these stay empty after polling, Copilot did not start even if the command returned success.
- The GraphQL `requestReviewsByLogin` fallback and REST `reviewers[]=copilot-pull-request-reviewer[bot]` can return success while `requested_reviewers` remains empty. Treat that as "not started"; check repository/organization Copilot Code Review settings, automatic review rulesets, or request Copilot manually in the GitHub UI.
- PowerShell treats bare `@copilot` as syntax; quote it as `'@copilot'` in `gh pr create --reviewer ...` and `gh pr edit --add-reviewer ...`.
- The `padosoft-laravel-flow` Copilot skill defines the full loop, not just the request command: open PR with `--reviewer copilot`, wait 60-180 seconds for CI, wait up to 15 minutes for Copilot, read PR review summaries, top-level comments, inline comments, and GraphQL `reviewThreads`, then fix/push/re-request until CI and Copilot converge.
- To know Copilot answered, require a visible Copilot-authored review or comment. Prefer entries whose `commit_id` matches the current `headRefOid`; `reviewDecision` alone is not enough because Copilot normally leaves comment reviews.
- After a fix push, Copilot may not re-review automatically unless review-on-push rulesets are enabled. Request/re-request with the same reviewer sequence, or use the GitHub UI re-request control when the API silently no-ops.
- Copilot review threads can remain non-outdated after nearby-line fixes; verify the current file content at the referenced path/line before resolving or declaring a comment addressed.
- Keep `docs/PROGRESS.md` as durable restart state rather than per-poll Copilot/CI history; detailed iteration logs belong in the PR, while reusable process discoveries belong in `docs/LESSON.md`.
- If the repo token has admin permission, Copilot Code Review can be enabled through a repository ruleset with rule type `copilot_code_review` and `review_on_push=true`. Target `~DEFAULT_BRANCH`, `refs/heads/main`, `refs/heads/task/*`, and `refs/heads/macro/*`; a broad `refs/heads/*` pattern did not match slash-containing branches in the branch-rules API. Verify with URL-encoded branch names such as `rules/branches/task%2Fbootstrap-agent-rules`, then push a new or amended commit to trigger automatic review.
- When Copilot is blocked by quota, budget, access, or prolonged non-response, use ChatGPT Codex Connector as automatic fallback. The working pattern from `padosoft/scalar-openapi-doc` PR #16 is to comment `@codex review`; the connector responds as `chatgpt-codex-connector[bot]` in PR reviews with a `Codex Review` body and `Reviewed commit:` marker. Requesting it as a GitHub reviewer is not required; `requested_reviewers` can remain empty while Codex reviews are present.
- Codex connector review caught that a lingering "Do not substitute Codex review for Copilot review unless explicitly requested" rule contradicted the automatic fallback. When adding fallback paths, remove or narrow older prohibitions in the same file so agents do not stall on conflicting instructions.
- Codex connector review caught that `docs/RULES.md` said MIT while the repository `LICENSE` file is Apache-2.0. Keep package metadata, README, Packagist, and process docs aligned to the checked-in license text.
- For Laravel 13 package testing, `orchestra/testbench:^11.0` is the compatible Testbench line; Packagist currently shows Testbench 11 requiring Laravel `^13.9.0` and PHP `^8.3`.
- Herd PHP 8.5 on this workstation reports `memory_limit=128M`; PHPStan level 8 can exhaust that limit even on W1. Use `vendor/bin/phpstan analyse --memory-limit=512M` for local gates unless the PHP ini is raised.
- This package is a library and `.gitignore` intentionally excludes `composer.lock`; validate and install locally, but do not add the generated lock unless the project policy changes.
- W1 Foundation has no UI surface. Vite and Playwright gates are not applicable until a milestone introduces UI/UX; still record that explicitly in `docs/PROGRESS.md` for each non-UI subtask.
- Codex PR review caught two W1 labeler guardrails: matching `tier_hints` entries must fail loudly when they reference an undefined tier, and host-specific preprint classification such as `arxiv.org` must win before generic DOI detection to avoid over-ranking preprints as peer-reviewed.
- Codex PR review caught a DTO shape issue: `list<>` fields must reject associative arrays with `array_is_list()` instead of normalizing values and silently changing invalid JSON-object input into a list.
- On this Windows checkout, merging a PR can re-materialize PHP/config/test files with CRLF and make `vendor/bin/pint --test` fail on `line_ending`. Keep `.gitattributes` in the repo with LF rules for PHP, JSON, XML, NEON, YAML, and Markdown before opening macro PRs.
- Codex macro PR review caught another DTO shape guardrail: use `array_key_exists()` when applying defaults so missing list/map keys can default to empty arrays while explicit `null` fails validation instead of being silently converted to empty data.
